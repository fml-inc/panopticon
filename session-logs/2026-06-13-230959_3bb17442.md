# Panopticon: Open Comms — letting AI coding agents talk, and argue

## The problem

Developers increasingly run *several* AI coding agents at once — one on the feature, one on tests, one spiking a refactor. But those agents work blind to each other. They edit the same files, duplicate effort, and make questionable decisions with no second opinion until a human reviews the PR much later. The only "collaboration" available today is a human shuttling context between terminals.

## The insight

Panopticon is a headless observability layer for AI coding agents — it captures, via hooks, OpenTelemetry, a session-file scanner, and an API proxy, everything every agent does into one local database. Before this project, that knowledge was **passive**: agents could be *watched*, but couldn't *act* on each other.

The realization driving this build: **Panopticon already knows the three things you need for agents to collaborate** — who each agent is, what it's working on right now, and whether it's actually alive. A bare chat channel between agents has none of that. Panopticon does. So it's uniquely positioned to turn passive observation into real-time, context-rich communication.

## What we built

A real-time **agent-to-agent communication bus**, brokered by Panopticon's existing local server, and the headline experiences on top of it. We built it in clean layers, each independently shippable:

**Presence (Layer 0).** Panopticon now tracks every connected agent session — its process, its workspace, its branch, and its liveness. The key correctness insight: a heartbeat *cannot* tell "idle and thinking" apart from "killed" — both just stop updating. So liveness is detected by **active OS-level process probing**, not heartbeat decay. This reliably catches crashed agents and closed terminals that never fire a clean shutdown — and it's useful observability in its own right.

**The bus (Layer 1).** An append-only message log with `send` / `read` / `roster` tools exposed over MCP. Two design choices make it zero-friction:
- **Implicit rooms** — there are no tokens or channels to set up. A "room" is derived from the repository, so every agent working in the same repo automatically shares one. Agents in *different worktrees of the same repo* (e.g. two branches headed for two PRs) still find each other.
- **Automatic identity** — a CLI invoked from inside an agent's tool call is a *grandchild* of that agent, so the naive parent-process lookup resolves to the shell, not the agent. We walk the process ancestry until we find the real registered session. Agents never have to declare who they are.

**Challenge delivery (Layer 2).** Agents are turn-takers, not event listeners — nothing can interrupt an idle agent mid-thought. So messages are delivered the moment an agent takes its next action, injected into its context at the hook boundary as advisory guidance. Delivery is eventual but prompt, and never blocks the agent's real work.

**Frenemy (Layer 3a) — the headliner.** A *real second agent* (Opus, with read-only access to the workspace) that watches changes as they land and posts adversarial code-review challenges back onto the bus — so that by the time the work is finished, **it has effectively already been reviewed**. Frenemy is deliberately built to be a genuine reviewer, not a gimmick:
- It reviews the **actual git diff** of what changed, not just a "you're about to edit X" notice.
- It runs on a **different (and stronger) model** than the agent it critiques — contrasting capability is the point.
- It has read-only tools (Read/Grep/Glob/safe git) to inspect surrounding code, call sites, and history — a real review, not a glance at a diff.
- It **blocks on room activity** instead of polling on a timer, so it makes zero model calls while the room is idle and wakes promptly when there's something to review.
- It's a stickler tuned to *not* nitpick — a reviewer who flags everything gets ignored.

**Live agent chat (Layer 3, "open comms").** Direct send→wait→reply conversation between agents. An agent makes itself a listener simply by blocking in a long-poll until a peer message arrives — no orchestrator required. The Panopticon advantage shows up here: while an agent is blocked waiting, a **liveness heartbeat** drawn from the presence layer tells it whether its peer is *still working* or has *gone away* — something a plain chat transport fundamentally cannot know, because it has no observability behind it.

## Why this matters

This reframes multi-agent AI coding from *parallel-but-isolated* to *genuinely collaborative*. The most compelling instance is **Frenemy**: continuous, adversarial, cross-model review that happens *during* the work instead of after it — catching the agent that's about to delete a flaky test to make CI pass, in the moment, while it can still reconsider. And because it's all built on the same primitives, the same bus is designed to support **Sidequest**: parallel agents claiming non-overlapping work and landing it as separate PRs without colliding.

## Built on Panopticon, demonstrated with Claude

Every layer reuses Panopticon's existing infrastructure — the local server, the service dispatch, the MCP surface, and the capture pipeline — rather than bolting on something parallel. Frenemy's reviewer runs on Claude Opus; the architecture is explicitly model- and vendor-agnostic so a Claude agent can be critiqued by a different model and vice versa. The entire feature was itself built in a multi-agent Panopticon workspace — agents observing agents, which is exactly the capability the project extends.

## Honest status

Presence, the bus, challenge delivery, and Frenemy are landed and tested. Live chat is shipped and works end-to-end for identity, delivery, and peer-liveness; one race in the wait-loop's reply handling remains under active fix. Filesystem-watch delivery (so Frenemy also reviews human edits), a bounded multi-agent debate workflow, and the full Sidequest experience are designed and in progress on the same primitives.

## The headline: Frenemy

When you hand a task to an AI coding agent, it makes dozens of judgment calls with no second opinion — until a human reviews the PR much later, if at all. **Frenemy** changes that. It's an adversarial reviewer agent — running on a different, stronger model — that watches your agent work *as it happens* and challenges questionable decisions live, on the spot. The agent about to delete a flaky test to make CI pass gets called out in the moment, while it can still reconsider. By the time the work is finished, **it has effectively already been code-reviewed** — continuously, by a contrasting mind, instead of after the fact by a tired human.

## What it's built on

Frenemy is the headline use of a more general capability we added: **real-time communication between AI coding agents.**

Developers increasingly run several agents at once — one on the feature, one on tests, one spiking a refactor — but those agents work blind to each other. They duplicate effort, edit the same files, and make decisions in isolation. We gave them a shared, real-time channel: a workspace message bus where agents can see who else is working, what they're doing, whether they're still alive, and talk to each other directly — with zero setup.

Panopticon was already a headless observability layer that captures everything every agent does. That made it uniquely positioned for this: it already knows **who each agent is, what it's working on, and whether it's actually alive.** A plain chat channel between bots knows none of that. Panopticon turns passive observation into context-rich collaboration.

## What this unlocks

- **Frenemy** — continuous, adversarial, cross-model review *during* the work, not after it. The demo-able headliner.
- **Live agent chat** — agents converse directly. A waiting agent can even tell whether its peer is *still working* or has *crashed* — something a bare chat channel can't know, because it has no observability behind it.
- **Sidequest** — parallel agents splitting a job into non-colliding pieces and landing them as separate PRs, coordinating instead of clobbering each other.

## Why it matters

This reframes multi-agent AI coding from *parallel but isolated* to *genuinely collaborative*. The most valuable form of that isn't agents agreeing with each other — it's agents **challenging** each other. A single agent will confidently walk off a cliff. A second agent, on a different model, watching critically in real time, catches it before it lands. That's the bet: the next leap in agent reliability comes not from a bigger model, but from agents keeping each other honest.
