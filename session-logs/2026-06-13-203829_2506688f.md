# Panopticon PR Review — #280

**Branch:** `layer3a-frenemy` (stacked on `layer2-recruitment-beacon`, +458/−5, 4 files)

## Summary

Adds `panopticon frenemy`: an adversarial sidecar process that polls the bus roster for live primaries in a room, pulls each primary's fresh hook activity (since a per-session cursor), feeds it to a headless `claude`/`codex` critic, and posts any `CHALLENGE:` back onto the bus directed at that primary — where Layer 2's `drainMessages` delivers it into the primary's next `PreToolUse` context. The driver is deterministic plumbing; the LLM is a stateless critic. Cleanly separated I/O via a `FrenemyDeps` seam makes `runFrenemyOnce` fully unit-testable.

The second commit is an unrelated-but-adjacent hardening of `drainMessages` (fail-closed backlog gate).

## File-by-file

**`src/frenemy/driver.ts`** (new) — Well-structured. `formatActivity`/`parseChallenge` are pure and tested; `runFrenemyOnce` threads all I/O through `deps`; the persona prompt is sensibly biased toward `SKIP`. Integration points verified against the codebase:
- `busRoster`/`hookTimeline`/`busSend` signatures match `httpPanopticonService` ✓
- `hook_timeline` returns DESC-ordered events; combined with the `since` ISO filter and the local `e.timestampMs > cursor` re-filter, the boundary (inclusive `since` vs. exclusive cursor) is handled correctly ✓
- `invokeLlmAsync` returns `string | null`; `parseChallenge(null)` handled ✓
- roster exclusions (`role !== "frenemy"`, `session_id !== FRENEMY_FROM`, `status !== "exited"`) match the `InstanceView` shape ✓

**`src/frenemy/driver.test.ts`** (new) — Good coverage: parse (incl. SKIP/empty/null/lowercase), activity formatting, cursor advance, no-re-critique, SKIP-advances-cursor-without-send, and the frenemy/exited exclusion. 6/6 pass locally.

**`src/cli.ts`** — Standard commander wiring for the `frenemy` subcommand; `--once` and loop paths both present.

**`src/hooks/ingest.ts`** — The drain fix. `sinceMs = getInstanceFirstSeen(sessionId) ?? nowMs` correctly fails closed: previously `?? undefined` dropped the backlog gate entirely, draining the whole room's broadcast history to a reader with no presence row. Confirmed against `readAgentMessages` (line 122–131): the gate only affects broadcasts — directed mail (`to_session = @toSession`) still always delivers, so the frenemy's directed challenges are unaffected. Threading one `nowMs` through both providers + `markDelivered` is a correct consistency improvement.

## Issues found

**1. Cursor advances *before* critique → transient LLM failures silently drop activity (moderate).**
`driver.ts:` the cursor is set to `Math.max(...fresh)` *before* `await deps.critique(...)`. If the critic times out, errors, or returns `null` (e.g. `invokeLlmAsync` returns `null` when the agent binary isn't detected, or on `LLM_TIMEOUT_MS`), that activity is permanently marked seen and never re-evaluated. For a frenemy this is "fail open" (miss a challenge), which is defensible — but a single critic timeout silently burns a real questionable action. Consider advancing the cursor only after a *successful* critic response (challenge or explicit SKIP), leaving transient failures to retry next pass.

**2. Ctrl-C latency in the loop (minor, UX).**
`createFrenemyLoop` sleeps via `await new Promise(r => setTimeout(r, intervalMs))` with no timer cancellation, and `stop()` only flips a flag. After SIGINT the process lingers until the current sleep (default 8s) or in-flight critique (up to the LLM timeout) completes. Acceptable, but an `unref`'d/cancellable timer would make Ctrl-C feel responsive.

**3. No validation on `--interval` / `--runner` (minor, robustness).**
`Number(opts.interval) * 1000` → `NaN` for non-numeric input, and `setTimeout(…, NaN)` fires immediately, turning the loop into a tight poll that hammers the service + LLM. `--runner` silently coerces anything ≠ `"codex"` to `"claude"`, so `--runner cluade` runs claude without warning. A small guard (reject `NaN`/non-positive interval, validate runner against the documented set) would prevent a footgun.

## Reinventing the wheel
None. The driver correctly reuses existing capture (`hook_timeline`), the bus (`busSend`/`busRoster`), `invokeLlmAsync`, and `resolveRoom` rather than republishing an activity feed or adding new state — consistent with the stated "driver = plumbing" design.

## Suggestions
- Address #1 (cursor-after-success) — it's the only change with real behavioral teeth.
- The `lookback`/`limit: 8` means if a primary emits >8 events between polls, only the newest 8 are critiqued and the cursor jumps past the rest. Fine as a design choice, but undocumented; a one-line comment would help the next reader.
- `formatActivity` renders events newest-first (DESC from `hook_timeline`); the persona says "recent actions" so it's not wrong, but reversing to chronological would read more naturally to the critic.

## Verdict
**Approve (comment-only).** Architecture is clean, the seam is genuinely testable, typecheck + tests pass, and the integration points all check out against the actual service/bus/presence code. The drain fix is a correct, well-reasoned hardening. None of the issues block merge; I'd recommend fixing the cursor-before-critique drop (#1) either here or in a fast follow, since it's the one that can silently swallow a real challenge.
