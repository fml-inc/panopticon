# Panopticon PR Review — `layer1-bus-primitive`

## Branch
`layer1-bus-primitive` (3 commits, +1367 lines, 19 files)

## Summary
Adds two related primitives in two layers:

- **Layer 0 — instance presence** (`src/presence/`): a server-wide registry of every connected agent session. Liveness is determined by *actively probing the recorded pid* (`process.kill(pid, 0)`), not heartbeat decay — so kills/crashes/closed terminals that never fire `SessionEnd` are still detected. A reaper loop (`~7s`) marks dead pids `exited (pid_dead)`, and a separate TTL prune bounds table growth. Surfaced via the `instances` MCP tool + CLI command.
- **Layer 1 — agent message bus** (`src/bus/`, `src/db/bus.ts`): an append-only `agent_messages` log scoped to a "room" (repo-granularity workspace). Supports broadcast/direct addressing, kind filtering, a `sinceId` cursor, and consume-once delivery (`delivered_at_ms` + `markDelivered`). Surfaced via `bus_send`/`bus_read`/`bus_roster` MCP tools and CLI commands. Gated `enableBusDelivery` flag is defined for the not-yet-built Layer 2 hook drain; the tables/tools work regardless.

The design is clean: pid-probe liveness is the right call and is documented honestly (pid-reuse limitation, wrapper-shell caveat, revival-on-false-reap, terminal `session_end`). Test coverage is genuinely good — 24 tests covering status derivation, reaper, revival, terminality, pruning, address filtering, and consume-once.

## Issues found

**1. `resolveRoom` is dead code in production (`src/bus/room.ts:19`).** It's exported and unit-tested, but the only non-test reference is a comment in `ingest.ts:565`. Production room resolution goes exclusively through `roomForSession` (reading `panopticon_instances.room`). Presence itself records the room as `repo ?? null` in `ingest.ts:638` — it does **not** call `resolveRoom`. So the function's docstring claim that it "intentionally returns the same value Layer 0 presence records" is aspirational, not enforced by shared code. This is fine as Layer 2 groundwork, but either wire it in or add a one-line note that it's intended for the upcoming cwd→room path; right now a reader can't tell it's unused.

**2. Doc comment in `readInstancesResult` overstates "always shown" (`src/presence/store.ts:1293`+).** The comment says *"Freshly-discovered-dead rows (status exited but ended_at_ms null) are always shown."* But the filter returns `false` for any exited row when `includeEnded === false`, including freshly-dead ones:
```js
if (i.status !== "exited") return true;
if (!includeEnded) return false;          // drops freshly-dead too
return i.ended_at_ms == null || nowMs - i.ended_at_ms <= window;
```
The "always shown" guarantee only holds under the default `includeEnded: true`. The behavior is arguably correct (`includeEnded:false` means "hide exited", and a freshly-dead pid *is* exited), but the comment is misleading and no test covers a freshly-dead row under `includeEnded:false`. Fix the comment, or add a test pinning the intended behavior.

**3. Explicit-room reads leak directed messages (`src/service/direct.ts:busRead`).** `busRead` only sets `toSession`/`excludeFrom` from `session_id`. If a caller passes `room` explicitly with no `session_id` (valid per the schema), both filters are undefined and the read returns *all* messages in the room — including ones addressed `to` a specific other session, and the room's own traffic. For CLI inspection (`bus-read --room`) this is probably intended, but it's an addressing-bypass worth a comment so a future Layer 2 caller doesn't accidentally rely on `bus_read` for privacy. Not a blocker.

**4. Minor: `Number(opts.since)` NaN handling (`src/cli.ts` bus-read).** A non-numeric `--since` yields `NaN`; since `typeof NaN === "number"`, the `id > @sinceId` clause silently matches zero rows rather than erroring or being ignored. Low impact (operator-only CLI), but a `Number.isFinite` guard would fail louder.

## Reinventing the wheel
Nothing flagged. `isPidAlive` correctly uses the standard `process.kill(pid, 0)` idiom with proper `EPERM`-means-alive handling. The store/service/transport layering matches existing conventions (`direct.ts`/`http.ts`/`transport.ts` mirror sibling tools). No reimplementation of existing repo resolution — it reuses `resolveRepoFromCwd`.

## Suggestions
- **Index alignment.** `readAgentMessages` always `ORDER BY id ASC`, but `idx_agent_messages_room` is `(room, created_at_ms)` — SQLite can't use it for the id-sort, so room reads filter-then-sort. Since `id` is monotonic with `created_at_ms` anyway, an index on `(room, id)` would serve both the filter and the order. The `idx_agent_messages_drain (to_session, delivered_at_ms)` index also won't help the actual undelivered-drain path, which filters `room + delivered_at_ms IS NULL` (room not in that index). Worth revisiting when Layer 2's drain query lands.
- The reaper is started with no `onReap` callback in `server.ts`. That's correct for this layer (no leases yet), but leaving a `// TODO: wire to claim expiry` would signal the intended Layer 1.5/sidequest hook.
