# Panopticon PR Review — #279

**Branch:** `layer2-recruitment-beacon` → base `layer2-bus-delivery` (stacked on #277)

## Summary

Two related changes to the Layer 2 agent message bus:

1. **Per-recipient delivery (group-chat fan-out)** — fixes a real bug where a single global `agent_messages.delivered_at_ms` column meant the *first* session to hit a hook consumed a broadcast for *everyone*. Delivery is now tracked per `(message_id, session_id)` in a new `agent_message_deliveries` table, so a room broadcast reaches every live session exactly once.
2. **Recruitment beacon provider** — `kind:'invite'` room broadcasts injected as advisory `additionalContext` ("📣 Open in your workspace…"), shown once per session within a 15-minute freshness window.

The design is clean: both features compose as providers behind `drainCoordinationContext`, gated by the pre-existing `enableBusDelivery` flag and `!isFrenemySession` exclusion at the call site.

## Issues found

Nothing blocking. Two minor points:

1. **Legacy-data re-delivery on upgrade (low risk, worth noting).** The new drain path gates on `undeliveredTo` (the new table), not the old `agent_messages.delivered_at_ms` column. On an existing DB, any message previously marked delivered via the old column has no row in `agent_message_deliveries`, so it would re-inject once after this lands. Directed challenges are most affected since the `sinceMs` gate doesn't suppress them (they're "mail"). Given Layer 2 is brand-new and likely has no real data yet, this is probably moot — but if any DB already carries delivered Layer-2 messages, expect a one-time re-inject. A backfill (`INSERT … SELECT id, to_session, delivered_at_ms FROM agent_messages WHERE delivered_at_ms IS NOT NULL AND to_session IS NOT NULL`) would close it if you care.

Still, the fallback silently changes behavior rather than failing closed. Consider defaulting `sinceMs` to `nowMs` (or the event timestamp) when presence is absent, so a session with no join record doesn't inherit the whole backlog.
