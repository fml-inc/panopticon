# PR 281 Review — Frenemy long-poll: block on room activity instead of fixed polling

**Branch:** `frenemy-long-poll` → base `layer0-instance-presence` (not `main`; the bus/frenemy stack was flattened onto layer0)
**Scope:** 10 files, +267/−23, single commit `9492393`

## Summary

Replaces the frenemy reviewer's fixed-interval poll (`setInterval`-style sleep, default 8s) with a server-side long-poll on room activity. A new in-process watermark/waiter registry (`activity-wait.ts`) lets a watcher block until a hook event newer than its cursor lands in the room. `ingest.ts` calls `noteRoomActivity` on every non-frenemy event; the loop long-polls `wait_for_activity`, settles briefly to batch a burst, then runs a pass. `--interval` is repurposed from poll period to settle window (8s→3s default). Wired through direct/http/transport service layers with a new `WaitForActivity` type pair.

The core idea is sound and the in-process primitive is clean. My concerns are about the **end-to-end feedback loop** and one **shutdown claim**, not the watermark logic itself.

## File-by-file

- **`src/bus/activity-wait.ts`** (new) — Solid. Watermark map + per-room waiter set. Immediate-resolve when `current > sinceMs` correctly closes the lost-wakeup race between passes. `unref()` on timers, snapshot iteration (`[...set]`) to avoid mutation-during-iteration, empty-set cleanup, and a `_resetActivityWait` test seam. Strict `>` (not `>=`) on `sinceMs` is right and prevents same-timestamp spin. No issues.
- **`src/bus/activity-wait.test.ts`** (new) — Good coverage of the primitive: immediate-resolve, wake-on-notify, not-newer, timeout, room-scoping, multi-waiter.
- **`src/service/direct.ts`** — `waitForActivity` clamps `timeoutMs` to `[1000, 28000]`. The 28s ceiling sits below the HTTP client's `TOOL_TIMEOUT = 30_000` (http.ts:109), so the long-poll always returns before the client cuts the request — the clamp claim checks out (2s margin).

## Issues found

### 1. (High) Self-wake feedback loop: the critic's own hooks can re-trigger the frenemy unless `PANOPTICON_FRENEMY_ROLE` is set — and nothing in the repo sets it

The critic is not an API call — `driver.ts:195` spawns a real `claude`/`codex` subprocess (`invokeLlmAsync`) with `--setting-sources user` and read-only tools, running in the **primary's worktree**. That subprocess fires its own Claude Code hooks (Read/Grep/`git diff`/etc.), which POST to the same server → `processHookEvent` → `noteRoomActivity(repo, …)` in the **same room**.

The only thing preventing that from waking the frenemy is `isFrenemySession(data)` (ingest.ts:717), which is true **only** when `data.role === "frenemy"`. That role is stamped by the hook handler from `process.env.PANOPTICON_FRENEMY_ROLE` (handler.ts:329), which reaches the critic subprocess via `cleanEnv()` copying `process.env` (llm.ts:113-120).

But **`PANOPTICON_FRENEMY_ROLE` is never assigned anywhere in the repo** — no CLI action, no install/bashrc/env.sh generation, no spawn env. Every reference is a *read* (handler.ts, ingest.ts comments). So the invariant holds only if the operator manually `export`s it before `panopticon frenemy`. If they don't:

- The critic's tool calls land as non-frenemy activity → `noteRoomActivity` wakes the long-poll → after the pass, `waitForActivity(sinceMs=watermark)` returns immediately (server watermark was bumped by the critic) → settle (3s) → another review → another critic → … a ~3s-paced review storm that never goes idle, plus the critic session shows up as a "primary" in `runFrenemyOnce` (role filter at driver.ts:226) and gets reviewed too.

Under the old fixed 8s poll this same gap only caused bounded, wasteful re-reviews; the long-poll converts it into a tight self-sustaining loop. That makes this PR the thing that turns a latent config dependency into a live failure mode.

**Fix:** make the marker self-enforcing rather than operator-dependent. Simplest is in the `frenemy` CLI action (cli.ts, before `createFrenemyLoop`):

```ts
process.env.PANOPTICON_FRENEMY_ROLE ??= "frenemy";
```

`cleanEnv()` then propagates it to every critic subprocess, so the critic's hooks are reliably skipped by `noteRoomActivity` and excluded from the roster. (`invokeLlmAsync` has no env-override param, so setting it on the parent process is the cleanest path.) Worth a focused test asserting a `role: "frenemy"` hook does **not** wake `wait_for_activity`.

### 2. (Medium) "Ctrl-C exits immediately even mid long-poll" looks inaccurate — the abandoned request socket keeps the process alive

`untilStop` only races the *promise*; it never aborts the underlying HTTP request. In `post()` (http.ts:42-105) the request socket is reffed and nothing calls `req.destroy()`/`abort()`/`socket.unref()`. The installed `process.on("SIGINT", …)` overrides Node's default terminate, and `parseAsync()` doesn't `process.exit` on success. So in the common case — idle, blocked on the 25s long-poll — Ctrl-C resolves `done` and returns from the action, but the in-flight socket remains an active handle and keeps the event loop alive until the **server** responds (clamped ≤28s). Repeated Ctrl-C just re-calls the no-op `stop()`. That's up to ~25s of apparent hang, contradicting the PR's claim.

**Fix:** abort the in-flight long-poll on stop (track the `ClientRequest` and `req.destroy()` it), or `unref()` the long-poll socket, or `process.exit(0)` after `await handle.done` in the CLI action. (I reasoned this from the code rather than running it live — worth confirming by Ctrl-C'ing an idle frenemy, but the mechanism is clear.)

### 3. (Low) One redundant review pass per burst with mid-review activity

After a pass, `watermark` is set to the activity ts that *woke* it (driver.ts), not to the newest activity seen during settle+review. If edits land during the settle/review window, the server watermark advances past `watermark`, so the next `waitForActivity` returns immediately and runs one extra pass (usually a SKIP). Harmless on its own — and the conservative choice avoids dropping reviews for edits that land after the diff is read — but worth a comment, since it compounds #1 when the marker is missing.

### 4. (Low) Test coverage gaps for the load-bearing invariant

- The integration "wakes when a hook lands" test relies on a 50ms `setTimeout` to register the waiter before posting — mildly racy under CI load (works, but a tighter handshake would be more robust).
- Nothing tests the negative: that a `role: "frenemy"` event does **not** wake a waiter. Given that this is the entire safety mechanism behind #1, it deserves an explicit test.

## Reinventing the wheel

None. The waiter/watermark primitive is purpose-built and doesn't duplicate anything in the bus or service layers; reusing `resolveBusRoom`/`roomForSession` keeps room resolution consistent with presence and Layer 1.

## Suggestions

- Land the `PANOPTICON_FRENEMY_ROLE ??= "frenemy"` default (#1) — it's the highest-leverage one-liner here and removes a silent dependency on operator env setup.
- Document the activity-wait state as **server-process-singleton** in the module header — it works precisely because `/hooks` ingestion and `wait_for_activity` share one process; a future move of ingestion off the server (or a second server) would silently break wakeups.

## Verdict

**Request changes.** The primitive and wiring are well done, but the end-to-end loop has a real self-triggering hazard (#1) that the long-poll makes acute, gated entirely on an env var nothing in the repo sets, and the prompt-shutdown claim (#2) doesn't appear to hold. Both have small, localized fixes. #3/#4 are polish.
