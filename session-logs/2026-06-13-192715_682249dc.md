# Panopticon Command Router

Route command-shaped Panopticon requests to either Panopticon MCP tools or the `panopticon` CLI. Treat `/panopticon <args>` and `$panopticon <args>` as the same command surface.

## Routing Rules

1. Parse the first token as the subcommand. If there is no subcommand, show concise help and prefer a lightweight `summary` or `status` lookup only if the user asked for current state.
2. Prefer MCP for read-only data queries because it returns structured, compact results.
3. Prefer the CLI for lifecycle, log, sync, prune, scan, install, uninstall, update, and doctor operations.
4. Use normal tool-approval and safety rules for write or destructive operations. Do not silently run `prune`, `sync reset`, `permissions apply`, `uninstall --purge`, or equivalent commands.
5. Keep output concise. Summarize large JSON/tool results instead of dumping everything unless the user asks for raw output.

## Issues found (all non-blocking)

1. **Seed-vs-stream race → duplicate feed rows (low).** At the bottom, `seed()` (async, not awaited) and `connectStream()` (synchronous `EventSource`) both start. A `message` arriving over SSE before the bus-seed query runs will be rendered live *and* re-rendered by the seed's `LIMIT 50` history — `applyMessage` has no id-based dedup, so you get a duplicate row at the boundary. Similarly the `instances` snapshot can `instances.set` over a fresher SSE delta (self-heals on next heartbeat). This is the one I'd actually fix: dedup `applyMessage` by `msg.id`, or `await seed()` before `connectStream()`.

2. **Burn-rate metric caveats (low).** `refreshSessionMeta` pulls `sessions` with `limit: 100`, so `totalSpend`/`burn`/`tokens` silently cover only the 100 newest sessions. Worse for the rate: when a *new* session enters that window between polls, its full historical cost is attributed to a single 15s interval and spikes `burnPerHr`; a window slide-out drops `sumCost` (clamped to 0 by the `Math.max`). Fine for a glanceable dashboard, but worth a comment so nobody trusts it as accounting.

3. **`toolHint` silently empty on large tool inputs (cosmetic).** The client calls `timeline` without `fullPayloads`, so `inputJson` is truncated to 500 chars server-side. `JSON.parse` of the truncated string throws and `toolHint` returns `""`, so hints just vanish for any tool call with >500-char input. Not a crash; consider requesting `fullPayloads` for the drawer or extracting the hint more defensively.

4. **`/ui` is unauthenticated and embeds the bearer token in the served HTML (low).** Any localhost client that GETs `/ui` receives the token, unlike `/api/*` which requires the bearer. This is mitigated by localhost binding plus same-origin policy (cross-origin DNS-rebind reads of the HTML body are blocked) and matches the stated localhost model — but it is a widening of the unauthenticated surface. Worth a one-line acknowledgment in the static.ts header comment.
