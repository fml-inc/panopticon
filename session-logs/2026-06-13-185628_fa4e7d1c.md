## Routing Rules

1. Parse the first token as the subcommand. If there is no subcommand, show concise help and prefer a lightweight `summary` or `status` lookup only if the user asked for current state.
2. Prefer MCP for read-only data queries because it returns structured, compact results.

# Panopticon PR Review

**Branch**: `layer1-bus-primitive` (base `main`)

## Summary

This branch lands two layered features behind the agent-coordination effort:

- **Layer 0 — instance presence** (`2207771`, `149934e`): a generic `panopticon_instances` registry of every agent session connected to the server, with liveness derived from *actively probing the agent pid* (not heartbeat decay). A reaper loop marks killed/crashed sessions `exited (pid_dead)`. Surfaced via a `instances` CLI command + MCP tool.
- **Layer 1 — agent message bus** (`7fb1116`, `dc63ce4`): an append-only `agent_messages` log scoped to a "room" (repo-granularity workspace). Send/read/roster exposed via CLI, MCP tools, and the HTTP exec/tool dispatch. MCP self-identifies the calling session via the `~/.claude/sessions/<ppid>.json` registry so bus tools work with zero arguments.

Bus *delivery* (auto-publishing hook activity and draining into hook context) is gated behind `PANOPTICON_ENABLE_BUS_DELIVERY` (default off); the tables/tools work regardless. Presence is always-on.

The work is well-structured, thoroughly commented (the design tradeoffs and limitations are documented in-code, which is excellent), and well-tested: 24 unit tests + 6 integration tests, all green, clean `tsc`.

## File-by-file

- **`src/db/schema.ts`** — two new `CREATE TABLE IF NOT EXISTS` tables + indexes appended to base schema. Idempotent, safe for existing DBs. `kind`/`room` as plain TEXT (no migration for new message types) is a good call.
- **`src/presence/store.ts`** — pid-probe liveness, COALESCE-fill upsert with reaper-revival semantics, terminal `session_end`. The `ON CONFLICT` revival logic is subtle but correct and well-commented.
- **`src/presence/reaper.ts`** — clean interval loop with `unref()`, injectable clock, `runOnce` for tests, `onReap` hook for downstream lease expiry. Errors swallowed + logged so a bad pass never crashes the server. Good.
- **`src/db/bus.ts`** — insert/read/markDelivered. Parameterized SQL throughout; transaction around `markDelivered`.
- **`src/bus/room.ts`** — `resolveRoom(cwd)` and `roomForSession(id)`, both resolving to repo granularity.
- **`src/mcp/server.ts`** — `resolveSelfIdentity()` + 4 tools.
- **`src/service/{direct,http,transport,types}.ts`** — bus-send routed through **exec** (write), read/roster/instances through **tool** (read). Correct separation.
- **`src/hooks/{handler,ingest}.ts`** — forwards `process.ppid` as `agent_pid`; records presence on every hook event, wrapped in try/catch so it never breaks ingest. Good defensive posture.
- Tests are strong: pid-reuse via spawn-then-reuse, revival, terminal exit, room scoping, address filtering, cursor paging, full HTTP wire path.

## Issues found

Nothing blocking. A few things worth a look, roughly in priority order:

1. **`bus_read` returns the *oldest* N messages, not the newest (`src/db/bus.ts:470`).** The query is `ORDER BY id ASC ... LIMIT @limit`. For cursor polling (`sinceId` advancing) this is exactly right. But for an *initial* read of a busy room with no `sinceId`, a new reader gets the oldest 200 messages and has to page forward to reach the present — the opposite of what a "read recent messages" tool description implies ("Read recent messages in your room"). Consider: when `sinceId` is absent, select the newest N (`ORDER BY id DESC LIMIT n`, then reverse), or document that a fresh reader should seed its cursor from `bus_roster`/a tail read. Today the tool name/description and behavior disagree at the margin.

2. **`bus_roster` / `instances` with no room resolved returns *all rooms* (`src/service/direct.ts:1690`).** `busRoster` falls back to `readInstancesResult({ room: undefined })`, which lists every instance across every workspace. The MCP tool says "List the agent sessions in *your* room." If `SELF_ROOM` resolution fails (the `~/.claude/sessions` coupling is best-effort) and the caller passes no `session_id`/`room`, they silently get a cross-workspace roster rather than an error or empty set. For a single-user same-host model this is low-impact, but it's a quiet scope widening — consider requiring a resolved room for `bus_roster`, or labeling the result as all-rooms.

3. **`bus_read` without a `session_id` disables address filtering (`src/service/direct.ts:1672`).** When `room` is given but `session_id` is not (e.g. CLI `bus-read --room X` with no `--session`), `toSession`/`excludeFrom` are both undefined, so the reader sees *all* messages including ones directed at other sessions. That's a reasonable "observer mode" for the CLI, but it means directed messages aren't actually private from a roommate who reads without identifying — worth a one-line doc note that `to` is addressing/filtering, not access control.

4. **`upsertInstance` bumps `last_seen_ms` even on an already-`session_end`-terminal row (`src/presence/store.ts:1268`).** The `MAX(...)` on `last_seen_ms` runs unconditionally while `ended_*` stay pinned. So a stray post-exit event advances `last_seen` on an exited row. Harmless (status stays `exited`, and the terminal test passes), but it's a small internal inconsistency — an exited row can show activity newer than its `ended_at_ms`.

5. **`resolveSelfIdentity` couples to an undocumented Claude Code file (`src/mcp/server.ts:693`).** Already acknowledged in the comment and fully guarded (falls back to explicit args on any failure), so this is fine as-is — flagging only so it's on the radar as a known fragility if Claude Code changes that path/format. `SELF_SESSION_ID`/`SELF_ROOM` are resolved once at module load, so a long-lived MCP server process that outlives a `cwd` change won't refresh — acceptable for the stdio-per-session model.

## Reinventing the wheel

Nothing flagged. The work reuses existing infra correctly: `resolveRepoFromCwd` (`src/repo.ts`) for room derivation — and I confirmed ingest's `repo` (top-priority `data.repository`, then `resolveRepoFromCwd`, `src/hooks/ingest.ts:234`) resolves to the same `.repo` string `resolveRoom` produces, so a room derived from a cwd and one recorded from hook events genuinely match. Presence plugs into the existing `getDb()`/schema/service/transport patterns rather than introducing a parallel store. Liveness via `process.kill(pid, 0)` is the standard idiom, correctly handling `EPERM`.

## Suggestions

- `BusReadResult.messages` is typed `unknown[]` (`src/service/types.ts:1839`). Since `AgentMessageRow` is exported from `db/bus.ts`, you could type it `AgentMessageRow[]` for callers (the HTTP boundary erases types anyway, but the direct service returns the real rows).
- `markDelivered` / `undeliveredOnly` are implemented and unit-tested but not yet wired into `busRead` — correct, since that's the Layer 2 hook drain. Just confirming it's intentionally dormant, not a dropped wire.
- Consider an index supporting `excludeFrom`/`from_session` if rooms ever get large; today `idx_agent_messages_room(room, created_at_ms)` plus small tables make it a non-issue.

No security or data-integrity concerns for the same-host single-user model.
