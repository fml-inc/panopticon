## Routing Rules

1. Parse the first token as the subcommand. If there is no subcommand, show concise help and prefer a lightweight `summary` or `status` lookup only if the user asked for current state.

## Panopticon PR Review

**Branch**: `layer0-instance-presence` (1 commit, +563 lines, all additions)

### Summary

Adds a generic, server-wide registry of every agent session currently connected to the Panopticon server (`panopticon_instances` table), exposed via a new `instances` MCP tool and `panopticon instances` CLI command. The distinguishing design choice is **active pid probing** rather than heartbeat decay: each hook event records the agent's pid (the handler's `process.ppid`), and a background reaper loop calls `process.kill(pid, 0)` on an interval to mark killed/crashed sessions `exited (pid_dead)` even when they never fire a clean `SessionEnd`. Clean status taxonomy: `active` (fresh heartbeat) / `idle` (stale but pid alive) / `exited` (ended or pid gone).

The layering is clean and the feature is well-isolated: a self-contained `src/presence/` module, best-effort ingest that can't break the hook pipeline, and an always-on reaper independent of scanner/sync. Tests are thorough (10 cases, all passing) and cover the genuinely tricky paths вЂ” idle-vs-exited disambiguation, false-positive-reap revival, COALESCE metadata merging.

### File-by-file

- **`src/presence/store.ts`** вЂ” Core registry. `isPidAlive`, `upsertInstance` (INSERTвЂ¦ON CONFLICT with `MAX(last_seen_ms)` and COALESCE fill), `endInstance`, `deriveStatus`, `readInstances`/`readInstancesResult`, `reapDeadInstances`. Well-documented, correct SQL.
- **`src/presence/reaper.ts`** вЂ” `setInterval` loop with `unref()`, injectable clock, `onReap` callback (not yet wired вЂ” reserved for sidequest lease expiry). Errors swallowed and logged. Good.
- **`src/hooks/handler.ts`** вЂ” Forwards `process.ppid` as `agent_pid` only if not already set. Reasonable.
- **`src/hooks/ingest.ts`** вЂ” Upsert on every event / `endInstance` on `SessionEnd`, wrapped in try/catch. Correctly uses server-side `timestampMs = Date.now()` (consistent with the reaper's clock вЂ” no skew risk).
- **`src/db/schema.ts`** вЂ” Table + two indexes via `CREATE TABLE IF NOT EXISTS`, auto-migrates on existing DBs through the unconditional `_db.exec(SCHEMA_SQL)`. Fine.
- **`src/service/*`, `src/mcp/server.ts`, `src/cli.ts`** вЂ” Standard wiring through the direct/http/transport service layers. Consistent with existing patterns.

### Issues found

**1. No retention/TTL on exited rows вЂ” unbounded growth + degrading roster (maintainability, should fix).**
`reapDeadInstances`/`endInstance` only set `ended_at_ms`; nothing ever deletes exited rows, and I confirmed no prune path touches `panopticon_instances` (`sync-prune.ts` and the scanner don't reference it). Combined with `readInstancesResult` defaulting `includeEnded` to **true**, every session ever connected accumulates forever and is returned by default. Over weeks this is thousands of rows and a roster dominated by long-dead sessions. Recommend either pruning exited rows older than a window (hook into `syncAwarePrune`), or having the default roster read return only recently-exited instances (e.g. `ended_at_ms > now - N min`) while keeping the row for history.

**2. `pid_start_hint` is dead scaffolding вЂ” pid-reuse blind spot (correctness, low probability).** `store.ts:174`
The column exists, the `InstanceUpsert` type carries it, but **no caller ever populates it and `isPidAlive` never consults it.** The comment in `store.ts` explicitly motivates it ("a stale heartbeat alone cannot distinguishвЂ¦"), yet the disambiguation it was meant to provide isn't implemented. Concrete failure: agent pid 1234 dies without `SessionEnd`; within the в‰¤7s reaper window the OS recycles 1234 for an unrelated process; the next reap probes 1234, sees it alive, and the instance is pinned `idle` forever. Low probability on a 7s timescale, but it's the one case the active-probe design is specifically claiming to handle. Either wire up the start-hint (capture process start time at upsert, compare on probe) or drop the column and note the limitation so it doesn't read as implemented.

**3. `process.ppid == agent` assumption is target-specific (robustness, minor).** `handler.ts:314`
Holds for Claude Code (hook handler is a direct child of the agent), but the registry is explicitly "generic across all targets." If another target invokes the handler under a wrapper/shell, `ppid` is either a short-lived shell (в†’ false `pid_dead` reap) or a long-lived shell (в†’ never dies). Best-effort and gated on `typeof !== "number"`, so non-fatal, but worth a comment that the pid is only meaningful when the handler is a direct child, and that targets without that property degrade to heartbeat-only.

**4. Minor: `deriveStatus` can report `status:"exited"` with `ended_reason: null`.** `store.ts:151`
A stale heartbeat + dead pid reads as `exited` immediately (before the reaper persists it), so a consumer keying off `ended_reason` to explain the exit sees `null`. Documented in the code comment and arguably correct ("immediately consistent reads"), but consumers should be aware status and `ended_reason` can momentarily disagree.

### Reinventing the wheel

Nothing significant. `isPidAlive` via `process.kill(pid, 0)` is the canonical Node idiom (no stdlib helper exists). The INSERTвЂ¦ON CONFLICT upsert matches the existing `upsertSession`/`upsertSessionCwd` conventions in this codebase rather than introducing a new pattern. The reaper loop mirrors `createScannerLoop`'s start/stop/unref shape. Good reuse of local idioms.

### Suggestions

- Address #1 before this sees real traffic вЂ” it's the one that bites silently over time.
- Consider that `upsertInstance` revives any ended row on *any* later event (`ended_at_ms = NULL` in the ON CONFLICT). Test `"a later heartbeat revivesвЂ¦"` frames this as intentional for false-positive reaps, but it also means a stray out-of-order event arriving after a genuine `SessionEnd` would resurrect the row as active. `SessionEnd` is normally terminal, so low risk вЂ” but if you want hard finality for `session_end` (vs. `pid_dead`), guard the revival on `ended_reason != 'session_end'`.
- `room` is currently just the repo (noted as Layer 1 TODO via `resolveRoom`). Fine as a placeholder; the index on `room` is already in place for it.

### Verdict

**Comment / approve-with-nits.** The core mechanism is sound, well-tested, cleanly layered, and the active-probe approach is the right call over heartbeat decay. None of the issues block correctness for the common path. I'd ask for **#1 (retention)** to be resolved or ticketed before this carries production load, and **#2** to be either implemented or honestly downgraded in the comments so the pid-reuse gap isn't masked by a column that looks wired but isn't.

No lingering intent questions вЂ” the commit message and inline comments fully explain the "why" (active probe vs. heartbeat, ppid forwarding, best-effort ingest), so I did not need to consult Panopticon/FML session history.
